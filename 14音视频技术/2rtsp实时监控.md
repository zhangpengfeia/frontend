# RTSP 实时流媒体协议
RTSP: 建立连接，请求视频，接收视频流
RTP: 实际传输视频和音频数据
RTCP: 质量控制、延时、丢包反馈
```
             控制
     ┌─────────────────┐
     │      RTSP       │
     └─────────────────┘
              │
              ▼
     ┌─────────────────┐
     │ RTP（传视频数据） │
     └─────────────────┘
     ┌─────────────────┐
     │ RTCP（质量反馈）  │
     └─────────────────┘
```
RTSP格式地址： rtsp://admin:123456@192.168.1.100:554/stream1
## 浏览器不支持 RTSP
> 绝大多数监控设备输出的实时视频，也就是rtsp，**浏览器根本不支持**。
> 因此必须经过“中间处理”，转成浏览器能接受的格式。
1. **协议复杂性：** RTSP 是一个有状态的、需要维持会话的协议，这与 HTTP 无状态、请求-响应的简单模型不同。
2. **传输层差异：** RTSP/RTP **最佳实践**是运行在**UDP**上（当然TCP也行），而现代 Web 的核心是 HTTP(S)/TCP。就算是使用WebSocket，也是一个HTTP `Upgrade`请求，之后在同一条 **TCP 连接**上实现全双工通信。毕竟浏览器使用UDP的话，安全性是无非保障的，毕竟如果网页能任意发送UDP包，将极大加剧DDoS攻击、端口扫描和网络滥用。只有在之后引入了WebRTC，才为浏览器**安全地引入UDP能力**
3. **安全模型：** Web 安全模型（同源策略等）很难直接套用在 RTSP 这种来自网络设备的流协议上。
4. **历史路径依赖：** 在监控系统发展的早期（2000年代初），互联网和浏览器并非主要目标。监控是封闭的、专业的系统，使用专用的客户端软件（如 NVR 软件、CMS客户端）。RTSP 是当时为实时性设计的成熟标准，自然成为行业选择。整个安防行业，从硬件芯片、设备固件、后端软件到开发工具，都是围绕 RTSP构建的庞大生态

## “转接”技术

正因为浏览器不支持 RTSP，催生了一系列“转接”技术。当你在浏览器中访问一个现代监控系统的网页时，无非就几种方案

**浏览器插件（已淘汰）：**

过去，监控厂商会提供 ActiveX 或 NPAPI 插件，让浏览器能直接播放 RTSP。但随着 Chrome、Firefox 等现代浏览器淘汰插件，这种方式已经被淘汰了

**纯前端解码**

利用 `WebAssembly` 技术，将 H.264 解码器编译成 wasm 在浏览器中运行，然后通过 WebSocket 接收原始的 RTP/H.264 数据包进行解码。不过这种方案，**技术要求高、延迟高、性能消耗大、兼容性还糟**。它要求浏览器做本不擅长的事情，并且只解决了视频解码问题，音频同步、网络适应性等问题依然棘手。这只是一种技术演示或特定场景的解决方案，无法成为行业标准。

**服务器端转码 / 转封装**
在后端服务器**实时地将摄像头的 RTSP 流拉取过来，然后将其转换成浏览器支持的标准格式**。
一般可以通过**FFmpeg**拉取摄像头的 RTSP 流，然后根据需要进行转码/转封装，当然有些厂商也有自己的SDK专门来进行处理。反正最后**浏览器支持**的格式其实也无非下面几种：

- **HTTP-FLV / WebSocket：** 早期常用，依赖 Flash
- **HLS（HTTP Live Streaming）：** 苹果推出的标准，将流切割成小的 `.ts` 文件片段，通过 HTTP 下载播放，不过延迟较高
- **MPEG-DASH：** 类似 HLS 的开放标准，国外常用
- **WebRTC**：服务器将 RTSP 流转码/转封装后，通过 **WebRTC 协议** 发送给浏览器，WebRTC 支持 UDP、低延迟（可做到 <500ms），并且是现代浏览器原生支持的标准


## Mediamtx 是什么？
Mediamtx（原名 rtsp-simple-server）是一个 **音视频协议网关 / 流媒体中间服务器**。
它的工作非常简单，但非常关键：
| 它能做的事情                                                 | 作用                         |
| ------------------------------------------------------------ | ---------------------------- |
| **接收摄像头推流（RTSP / RTMP / WebRTC / HLS / SRT / UDP ...）** | 摄像头 → Mediamtx            |
| **把流转成多种格式再输出**                                   | 提供给网页、APP、NVR 等使用  |
| 支持 **转协议、转码、录像、鉴权、回放、跨网络分发**          | 让摄像头能被安全、稳定地播放 |

也就是说：

```yaml
摄像头（RTSP） → Mediamtx → 用户浏览器（HLS / WebRTC）
```
### ① 摄像头采集视频
摄像头拍摄实时画面 → 编码成 H.264/H.265 → 打包成 RTSP / RTMP 协议流
大部分商用摄像头最终提供 **RTSP** 地址
### ② 把流送到流媒体服务器
摄像头把 RTSP 推到 Mediamtx：rtsp://192.168.1.88:554/mystream
### ③ 前端播放
当然前端可以自行选择使用HLS，WebRTC或者HTTP-FLV等等实现播放
