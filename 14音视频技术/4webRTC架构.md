# WebRTC架构
WebRTC 并不是某一个“协议”，而是**一整套技术栈**：
- **W3C**：负责 JS API 标准
  - `getUserMedia`
  - `RTCPeerConnection`
  - `RTCDataChannel`
- **IETF**：负责底层通信协议
  - ICE / STUN / TURN
  - RTP / RTCP
  - DTLS / SRTP

**2016年**：WebRTC被正式确定为W3C和IETF标准
## WebRTC 要解决的核心问题
### 超低延迟（实时）
- 视频会议
- 在线课堂
- 远程协作
- 游戏语音
目标延迟：**端到端 200ms 以内**，这是 HLS / DASH 永远做不到的
### 安全是强制的
WebRTC **强制加密**：
- 信令：HTTPS / WSS
- 媒体：
  - DTLS
  - SRTP
### 不只是音视频，还有数据
WebRTC ≠ 视频通话
它还提供：
- **RTCDataChannel**
  - 文本
  - 二进制
  - 文件
  - 游戏状态同步
> WebRTC 其实是一个 **通用的低延迟通信框架**

### 跨平台
一个完整的音视频直播架构除了实现基本的输入输出模块，还要考虑**跨平台的问题**。只有在各个平台上都能实现音视频的互通互联，才能称得上是一个合格的音视频直播客户端架构。所以至少应该支持Windows、Mac、Android以及IOS四个终端。当然，如果还能够支持Linux端和浏览器就更好了。
### 插件化管理

对于音视频直播客户端来说，我们不但希望它可以处理音频数据、视频数据，还希望它可以分享屏幕，播放多媒体文件。共享白板等等。此外，即使是处理音视频，我们也希望它可以支持多种编码格式，比如音频除了可以支持Opus、AAC外，还可以支持G.711/G.722，iLBC、Speex等等，视频除了可以支持H264外，还可以支持H265、VP8、VP9、AV1等等，这样才能应用得更加广泛。
### 其他
除了上面的几点外，其实还有很多工作要做
- 音视频不同步
- 回音
​	这个主要指的是与其他人进行实时互动的时候，还可以听到自己的回声。在实时音视频通信中，不光有回声的问题，还有噪声、声音过小的问题，我们一般统称为**3A问题**。这些问题都是非常棘手的，需要专门的回音消除算法来进行处理
- 音视频的实时性
​	要进行实时通信，网络质量尤为关键，虽然常用的TCP有一套完整的保证网络服务质量的方案，但是它在实时性方面表现又不佳。换句话说，TCP是以牺牲实时性来保证网络服务质量的方案，而实时性优势音视频实时通信的命脉，这就导致TCP不能作为音视频实时传输的最佳选择。因此为了保证实时性，一般情况下实时直播应该首选UDP。但是UDP又不稳定，所以这样一来，又必须自己编写一套网络控制的算法以保证网络质量
此外，还有网络拥塞、丢包、延时、抖动、混音......等等问题。所以自研一套音视频直播客户端是十分有难度的。

# 信令服务器
建立连接之前，浏览器之间没办法直接传递数据。
需要服务器中转，在浏览器之间传递这些数据，这个服务器就是信令服务器。

STUN服务器：用于获取公网IP地址
TURN服务器：部署在公网，可以转发两个端点音视频数据

## SDP
Session Description Protocol
会话描述协议,用于双方建立WebRTC连接前交换信息和协商
双方交换能力信息，达成一致兼容，从而决定如果建立WebRTC连接

## ICE
Interactive Connectivity Establishment
交互式连接建立:解决连接链路的选择，那条网络路径最好最优

ICE Candidate:ICE候选，把电脑上所有网络路径列出来，每一个路径叫ICE Candidate
局域网地址，STUN服务器，TURN服务器

# 多方通信架构
Mesh:点对点，多个终端直接两两连接
MCU:点对控制单元，每个客户端和中心的MCU服务器连接，服务端负责混流，客户端只需要推一路流，收一路合成流，服务器压力大
SFU:一个服务器和多个终端组成，服务器不混流，只负责根据订阅关系选择性转发媒体流，连接数多，消耗带宽大
